# 钉钉通讯录同步 · 手把手教程

> 把钉钉里的部门和员工同步进 Kimpo，并让员工能用钉钉扫码登录。
> 建议先读[总览](./README.md)了解「认领 / 权威模式 / 工号 / 密钥安全」这几个概念。

本教程分两大段：**A. 在钉钉侧准备** → **B. 在 Kimpo 控制台配置**。

---

## A. 在钉钉侧准备（约 5 分钟）

你需要一个"钉钉自建应用"，并给它开通"读通讯录"的权限，然后拿到 3 样东西：**企业 ID、应用 Client ID、应用 Client Secret**。

> ⚠️ 你必须是这个钉钉企业的**管理员**。如果还没有企业，可以在钉钉里免费创建一个团队/企业，你自动成为管理员。

### A1. 创建自建应用

1. 用管理员账号登录钉钉开放平台 👉 https://open.dingtalk.com/
2. 点右上角 **后台管理 / 开发者后台**，进入左侧菜单 **应用开发 → 企业内部开发**。
3. 点右上角 **【创建应用】**，填应用名称（如"Kimpo 通讯录同步"）、描述、图标，点 **【保存】**。

> 💡 **踩坑提醒**：钉钉近年改过菜单和叫法。老教程里说的「企业内部应用 → 创建自建应用」现在是「**应用开发 → 企业内部开发 → 创建应用**」。

### A2. 拿凭证：Client ID / Client Secret

保存后进入应用详情页，找到 **「凭证与基础信息」**，复制这两个值：

- **Client ID**（页面上会标注"原 AppKey"）
- **Client Secret**（页面上会标注"原 AppSecret"）

> ⚠️ **踩坑提醒**：钉钉正在把凭据统一成 OAuth 术语——原来的 **AppKey 现在叫 Client ID**，**AppSecret 现在叫 Client Secret**。你在页面上找不到"AppKey"是正常的，认 **Client ID** 就对了。

**企业 ID（CorpId）**：在开发者后台首页或企业信息里能看到，形如 `dingxxxxxxxx`。

### A3. 开通"读通讯录"权限

1. 在应用内左侧点 **【权限管理】**。
2. 在搜索框输入 **「通讯录」**。
3. 勾选所有**读**类权限点（部门信息读、部门成员读、成员详情读；如果以后要同步手机号，再加"个人手机号信息"）。
4. **授权范围**选 **全部员工 / 全部部门**。
5. 点 **【批量申请】**。自建应用你就是管理员，权限**即时生效**、无需审核。

### A4. 关于服务器 IP 白名单（可能遇到）

- 如果创建应用时能选「**测试应用**」，**选它**——测试应用不受服务器 IP 白名单限制，最省事。
- 如果没有测试应用选项（应用按正式处理），钉钉会要求把**调用方服务器的公网出口 IP** 加进白名单。请让部署 Kimpo 的运维同事，把 Kimpo 服务器的公网出口 IP 填进应用的「服务器出口 IP / IP 白名单」。否则测试连接会报 `ip not in whitelist`。
- 应用**无需发布上线**，就能被服务端调用来读通讯录。

### ✅ A 段产出（拿去 B 段用）

| 项 | 从哪拿 | 举例 | 敏感？ |
|---|---|---|---|
| 企业 ID（corp_id） | 开发者后台首页 | `dingxxxxxxxx` | 否，可填界面 |
| 应用 ID（client_id） | 凭证与基础信息 → Client ID | `dingsfxxxxxx` | 否，可填界面 |
| 应用密钥（client_secret） | 凭证与基础信息 → Client Secret | 一长串 | ✅ **敏感，走环境变量** |

## B. 在 Kimpo 控制台配置

### B0. 先把密钥交给运维设成环境变量

把 A 段拿到的 **Client Secret** 交给部署同事，设成 Kimpo 服务器的一个环境变量，例如：

```
DINGTALK_APP_SECRET = <你的 Client Secret>
```

你自己记住这个**变量名**（`DINGTALK_APP_SECRET`）就行——一会儿在界面里只填这个名字，不填密钥本身。

> 详见[总览 §5 密钥安全铁律](./README.md#5-一条安全铁律密钥不落库不填明文)。

### B1. 新建目录源

1. 进入 **控制台 → 外部目录** 页签，点 **【新建目录源】**。
2. 按下表填写：

| 表单项 | 填什么 |
|---|---|
| **目录类型** | 选「钉钉」 |
| **名称** | 随便起个好认的名，如"总部钉钉" |
| **企业 ID（corp_id）** | 填 A 段的企业 ID |
| **应用 ID（client_id）** | 填 A 段的 Client ID |
| **凭据引用** | 填**环境变量名** `DINGTALK_APP_SECRET`（不是密钥本身！） |
| **权威模式** | 选「仅补全本地空字段」（推荐，最安全） |
| **匹配策略** | 选「自动绑定已有对象」（开启认领，避免建重复） |
| **登录来源码** | 保持默认 `dingtalk`（用于将来 SSO 配对钉钉登录插件） |
| **锚点部门** | 一般留空（默认挂到公司根）。除非你想把外部部门统一挂到某个指定部门下 |

3. 点保存。

### B2. 测试连接

在目录源上点 **【测试连接】**。成功会显示：

> 连接成功：部门 X · 用户 Y

- 这一步验证：密钥能从环境变量读出来 + 凭据正确 + 通讯录读权限已开通。
- 如果失败，看[下方常见问题](#常见问题)。

> 小知识：测试连接显示的部门数是一个"快速探测"（只数根部门的直接下级），可能比真实全量少。**以「同步预览」的结果为准。**

### B3. 同步预览

点 **【同步预览】**。Kimpo 会去钉钉拉取完整部门树和员工，算出要做哪些改动，生成一批**差异**（这一步**不写数据**）。稍等几秒，状态变成"已预览"。

### B4. 差异审阅 & 认领

进入 **【差异审阅】**，你会看到一条条差异，类型有：**新建 / 更新 / 认领绑定 / 成员关系** 等。

- 如果 Kimpo 里**已经手工建过**某个同名部门/人，想让它们合并：对那条"新建"点 **【绑定到既有部门…】** 或 **【绑定到既有用户…】**，选中你已有的对象——它就变成「认领绑定」，应用后不会建重复。
- 认了父部门后，它的子部门会自动跟着认，你只需认最顶层一次。

### B5. 应用

确认无误，点 **【应用已选】**。这一步才真正写入 Kimpo。完成后显示：

> 已应用 X，跳过 Y，失败 Z，冲突 W

去 **控制台 → 部门用户** 看看，钉钉的部门树和员工应该已经进来了。

### B6. 员工工号会是什么样

- 如果员工在钉钉里**填了工号**（钉钉的 `job_number` 字段），Kimpo 的工号就是它，例如 `A1001`。
- 如果**没填**，Kimpo 会自动生成一个系统工号，形如 `U000123`，你以后可以在部门用户里改。
- 那串看不懂的钉钉 userid **不会**变成工号——它只作为登录身份存在后台。

### B7. 以后再同步

钉钉那边有人入职/离职/调岗后，回到目录源重复 **测试连接（可选）→ 同步预览 → 应用** 即可。Kimpo 只处理变化的部分，已同步的不会重复建。

## 常见问题

| 现象 | 原因 & 解决 |
|---|---|
| 测试连接报 `invalidClientIdOrSecret` | Client ID/Secret 填错，或环境变量没设对。核对凭据、确认运维已设好环境变量且变量名与"凭据引用"一致 |
| 测试连接报 `ip not in whitelist` | 正式应用需要 IP 白名单。让运维把 Kimpo 服务器公网出口 IP 加进应用的 IP 白名单，或改用"测试应用" |
| 测试连接报权限不足 | 通讯录读权限没勾全或没【批量申请】。回 A3 重新申请，授权范围选"全部" |
| 预览部门数比测试连接多 | 正常。测试连接是浅探测，预览才是全量。以预览为准 |
| 同步进来的人工号是一串数字 ID | 你用的是旧版本行为。新版本已修正：真实工号走 `job_number`，没有则生成 `U000xxx`，技术 ID 不再当工号 |
