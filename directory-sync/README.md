# 外部目录同步 · 新手上手指南

> 面向**平台管理员 / 实施人员**的手把手教程：把钉钉、企业微信、飞书、LDAP/AD 里的部门和员工，安全地同步进 Kimpo，并让他们能用原有身份登录。
> 不需要你懂代码，也不需要读宿主源码。跟着一步步点即可。

---

## 1. 这是什么、能帮你做什么

企业里"谁在哪个部门、谁是谁"这套通讯录，通常已经维护在**钉钉 / 企业微信 / 飞书**，或者传统的 **LDAP / Active Directory** 里。你不想在 Kimpo 里再手工把几百号人重新录一遍。

**外部目录同步**就是干这件事的：

- 把外部通讯录的**部门树**和**员工**，成建制地同步进 Kimpo；
- 员工离职、换部门，下次同步会**自动跟着变**；
- 同步进来的人，可以直接用**钉钉扫码 / 飞书 / 企业微信 / 域账号**登录 Kimpo（单点登录 SSO），不用再单独设一套密码；
- 如果 Kimpo 里已经手工建过一些部门和人，同步会**认领**它们、和外部对象合并成同一个，而**不是**建一堆重复。

一句话：**外部维护通讯录，Kimpo 这边一键对齐。**

## 3. 先搞懂 4 个概念（很重要，能少踩坑）

### 3.1 认领：不会建重复

假设你的 Kimpo 里已经手工建了个"研发部"，钉钉里也有个"研发部"。同步时 Kimpo 会**认领**——把钉钉那个绑定到你已有的"研发部"上，而不是又建一个。

- **自动认领**：外部对象和 Kimpo 已有对象编码相同、或同名且父部门已绑定，同步会自动把它标成「认领绑定」。
- **手动认领**：在「差异审阅」界面，你可以对某条"新建"手动点【绑定到既有部门…】/【绑定到既有用户…】，把它认到一个现有对象上。
- 认领只建**关系**，不建重复对象；认了父部门后，它下面的子部门会**自动逐层跟着认**。

### 3.2 权威模式：冲突时听谁的

同一个人，外部叫"张三"、Kimpo 里被改成了"张三（组长）"，同步时该听谁的？由**权威模式**决定：

| 权威模式 | 含义 | 建议 |
|---|---|---|
| **仅补全本地空字段** | 只填 Kimpo 这边是空的字段，绝不覆盖你改过的内容 | ✅ **推荐**，最安全 |
| 仅记录不覆盖 | 只记录差异、完全不动 Kimpo 现有数据 | 想先观察时用 |
| 外部覆盖本地 | 外部数据强制覆盖 Kimpo 本地改动 | ⚠️ 会冲掉你的手工修改，谨慎 |

### 3.3 工号：技术 ID 不当工号

外部系统里每个人有两类"号"，Kimpo 分得很清楚：

- **技术 ID**（钉钉 userid、LDAP 的 uid 等）：系统分配、不可改，是给机器和登录用的 → Kimpo 存在**登录账号**里，**不会**当成工号。
- **真实工号**（钉钉 `job_number`、LDAP `employeeNumber`、飞书 `employee_no`）：HR 给的员工编号 → Kimpo 存进**工号（user_no）**。
- 如果外部**没填**真实工号，Kimpo 会**自动生成**一个系统工号（形如 `U000123`），管理员以后可以改。**绝不会**把那串看不懂的技术 ID 塞给你当工号。

### 3.4 SSO：同步的人怎么登录

同步进来的员工，Kimpo 会顺手给他建一个**登录账号**（记录他在外部的身份）。只要你**同时启用了对应的登录插件**（钉钉登录 / 飞书登录 …），他扫码登录时就会**落到这个已同步的账号**上，不会分裂出两个号。

> 登录插件的启用是**另一件事**（不在本教程范围）。本教程负责把人同步进来并备好登录账号；扫码直登还需要管理员启用对应登录方式。

## 4. 通用操作流程（四家都一样）

进入 **控制台 → 外部目录** 页签，流程都是这 6 步：

```
①新建目录源  →  ②测试连接  →  ③同步预览  →  ④差异审阅（认领）  →  ⑤应用  →  ⑥（以后）再同步
```

1. **新建目录源**：选类型（钉钉/企微/飞书/LDAP），填企业 ID、应用 ID、凭据引用等（各家字段见对应教程）。
2. **测试连接**：点【测试连接】，成功会显示"连接成功：部门 X · 用户 Y"。这一步是验证你的凭据和权限对不对。
3. **同步预览**：点【同步预览】，Kimpo 去外部拉取并算出"要新建/更新/认领哪些"，生成一批**差异**。**预览不改数据**，只是给你看。
4. **差异审阅（认领）**：逐条看差异，需要认领的手动【绑定到既有部门/用户】。
5. **应用**：确认无误后【应用已选】，这一步才真正写入 Kimpo。会显示"已应用 X，跳过 Y，失败 Z，冲突 W"。
6. **以后再同步**：外部有人事变动后，重复 ②~⑤，Kimpo 只会处理**变化的部分**（增量），已同步的不会重复建。

> 每次同步任务都会留在【任务历史】里，可回看。

## 5. 一条安全铁律：密钥不落库、不填明文

各家都要一个**密钥**（钉钉的 Client Secret、飞书的 App Secret、LDAP 的绑定密码…）。Kimpo 的设计是：

- **密钥值绝不填进界面、也不存进数据库**；
- 你把密钥值设成**服务器的一个环境变量**（例如 `DINGTALK_APP_SECRET`）；
- 在 Kimpo 目录源表单的「**凭据引用**」里，只填这个**环境变量的名字**（不是密钥本身）；
- 运行时后端从环境变量里读出真实密钥去调用外部接口。

界面里那句提示说得很清楚：

> 「这里填的是保存密钥的环境变量名，不是明文密钥本身；密钥值由后端从环境变量解析。」

**怎么设环境变量**通常由你的运维在启动 Kimpo 服务时配置（联系部署同事）。设好后把"变量名"填进凭据引用即可。
