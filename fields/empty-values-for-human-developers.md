# 字段空值：Kimpo 为什么"空就是空"，而不是填 0 —— 面向人类的讲解

> 读者：模板设计者、插件开发者，尤其是**用惯了同类产品的老手**。
> 读完你会明白：两种空值方案到底差在哪、Kimpo 为什么选了现在这条路、它的代价是什么，以及你原来的习惯要怎么平移过来——大部分场景其实一行都不用改。

---

## 1. 两种方案：先讲清楚分歧点

用户填一张表单，"数量"这一格没有填。系统里该存什么？业界有两个流派：

### 老模式：哨兵值（同类产品的普遍做法）

没填？那我替你填一个"无害"的默认值：

| 类型 | 没填时存 |
|---|---|
| 字符串 | `""`（空字符串） |
| 数值 | `0` |
| 日期 | `1900-01-01` |

好处立竿见影：**任何公式、任何比较、任何函数都不会遇到"空"这个概念**。`LEN` 不报错，`数量 × 单价` 永远算得出数，日期永远可以比大小。设计者和用户从头到尾不需要知道"null"是什么。

### 新模式：诚实空值（Kimpo 的做法）

没填？那就是**没填**。数据库里、数据权威层（Mirror，代号 艾莉亚/Aria）里、接口里，这一格都是"空"（null）——一个和 `0`、`""`、任何日期都不同的状态。

"空值参与计算会不会到处报错？"——不会。因为 Kimpo 同时约定了一整套**空值参与运算的规则**（见第 4 节），空值在公式里会按上下文自动"扮演"合适的角色：算术里当 0，拼接里当空文本，判断里当"否"。**你在公式里的体验和老模式几乎一样，但账本上的数据是诚实的。**

一句话概括分歧：

> 老模式把"方便计算"提前焊死在**存储**里；Kimpo 把它推迟到**计算的那一瞬间**。数据只记录事实，方便是引擎的责任。

## 2. 一个故事看懂差别：仓库盘点

三个仓库上报库存：A 库报了 120，B 库报了 **0**（真的空了），C 库**忘了填**。

**老模式下**，C 库的空格被系统悄悄写成 0。月底报表上：

- 平均库存 = (120 + 0 + 0) ÷ 3 = **40** —— 被一个"假 0"拉低了；
- "库存为 0 的仓库"清单里出现 B 和 C —— 但 C 到底是空了还是没盘？**没人知道，数据里已经找不回这个信息了**；
- 老板问"哪些仓库还没交盘点？"——查不出来，因为"没交"这个事实在落库那一刻就被抹掉了。

**Kimpo 模式下**，C 库那格是空：

- 平均库存 = (120 + 0) ÷ 2 = **60** —— 空值不进分母（和 SQL、Excel 的 `AVERAGE` 行为一致）；
- "库存为 0"筛出来只有 B；"库存**为空**"筛出来是 C——两个问题，两个答案，都对；
- 公式 `=库存 × 单价` 对 C 库照样算（空当 0，得 0），**日常使用毫无障碍**。

日期的版本更扎眼：老模式下没填的"合同签订日"是 1900-01-01——它会出现在打印件上、排序时排在所有单据最前面、掉进"签订日早于 2000 年"的筛选里。每一个用过老模式产品的实施顾问，都清洗过这种"上个世纪的合同"。

## 3. 现方案的优点与缺点（诚实版）

### 优点

| # | 优点 | 说明 |
|---|---|---|
| 1 | **信息不丢失** | "填了 0"和"没填"是两个业务事实，永远可区分、可追问、可考核（如"未盘点仓库"） |
| 2 | **统计天然正确** | 平均/最小/计数自动跳过空值，不需要在每个报表里手工剔除假 0 |
| 3 | **报表干净** | 空就显示空白。不会打印出 1900-01-01，不会满屏无意义的 0 |
| 4 | **必填校验可靠** | "没填"有明确判据。老模式下 `0` 到底是用户填的还是系统垫的，校验器自己都说不清 |
| 5 | **和标准世界接轨** | 数据库（SQL NULL）、Excel（空单元格）、外部系统 API（JSON null）本来就是这套语义，数据进出零翻译 |

### 缺点（不回避）

| # | 缺点 | 缓解 |
|---|---|---|
| 1 | **多一个概念** | 设计复杂模板时，你偶尔需要意识到"空 ≠ 0"。缓解：日常公式无感（规则对齐 Excel），只有显式判空时才需要 `ISBLANK` |
| 2 | **"不等于"的意外** | 筛选"金额 ≠ 100"**不包含**没填金额的单据（空值不参与任何比较）。这是 SQL/Excel 的标准行为，但第一次遇到会愣一下 |
| 3 | **空日期不能参与运算** | `空日期 + 30 天` 得到的还是空，不是某个日期。这其实是优点（假日期更危险），但依赖"日期恒有值"的老公式需要检查 |
| 4 | **插件开发者要处理 null** | 读字段可能读到空，代码里要有分支。SDK 的 `TypedValue` 与"是否有值"标志已把这件事显式化，照契约写即可 |

## 4. 空值运算规则速查表

心法只有一句：**"把它当成 Excel 的空单元格"**。你在 Excel 里怎么预期，Kimpo 就怎么表现。

### 空值在公式里的表现

| 场景 | 结果 | 例子 |
|---|---|---|
| 算术 | 当 **0** | 数量为空：`=数量 × 单价` → 0；`=数量 + 1` → 1 |
| 文本函数/拼接 | 当 **空文本** | 备注为空：`=LEN(备注)` → 0；`="编号-" & 数量` → `"编号-"`（注意：**不是** `"编号-0"`） |
| 逻辑判断 | 当 **否（FALSE）** | `=IF(是否加急, "急", "普通")` 未勾选 → "普通" |
| 日期函数 | **结果也为空**（不报错、不当 1900 年） | 签订日为空：`=DATEDIF(签订日, 今天)` → 空 |
| 显式判空 | 用 **`ISBLANK(x)`** | `=IF(ISBLANK(库存), "未盘点", 库存)` |

### 空值在统计里的表现

| 函数 | 空值处理 | 全为空时 |
|---|---|---|
| 求和 SUM | 跳过 | 0 |
| 平均 AVG | 跳过（**不进分母**） | 空 |
| 最大/最小 | 跳过 | 空 |
| 计数 COUNT(字段) | 不计 | 0 |
| 行数 COUNT(*) | 计（数行不看字段） | — |

### 空值在筛选里的表现

| 筛选条件 | 空值单据是否命中 |
|---|---|
| 等于 / 不等于 / 大于 / 小于 / 包含… | **一律不命中**（连"不等于"也不命中） |
| **为空** | 命中（这是找空值的唯一入口） |
| **不为空** | 不命中 |

### 公式和表达式是两个世界（进阶，但很重要）

上面"空值在公式里的表现"说的是**单元格公式**（你在模板格子里写的 `=数量*单价`）。Kimpo 里还有另一处会写"式子"的地方：**取数/动作里的表达式**（计算列、写值表达式、触发条件）。两处对空值的态度刻意不同：

| | 单元格公式 | 取数/动作表达式 |
|---|---|---|
| 世界观 | **Excel 世界**：空自动扮演角色 | **数据库世界**：空诚实传播 |
| `数量 × 单价`，数量没填 | 得 **0** | 得**空**（未知 × 单价 = 未知） |
| 拼接 `"编号-" & 编码`，编码没填 | `"编号-"` | `"编号-"`（拼接两边一致，跳空） |
| 日期没填参与日期函数 | 结果空 | 结果空（一致） |
| 想要"没填按 0 算" | 天然如此 | 显式写 **`IFNULL(数量, 0)`**（函数面板里叫「空值默认」） |
| 判空 | `ISBLANK(字段)` | 条件里选「为空」 |

为什么不统一成 Excel 那套？因为表达式的产物是**数据**（算出来要写回字段、进报表）。表格公式里"没填当 0"是给你看的便利；但数据管道里替你捏造一个 0 写进账本，就等于把老模式的哨兵值从后门放回来了——"运费 + 货款(未知)"的诚实答案是"未知"，不是"= 运费"。想兜底，`IFNULL` 写出来，人人看得见。

### 各类型的"空"长什么样

| 类型 | 什么算空 | 特别说明 |
|---|---|---|
| 文本 | 没填，或只输入了空格 | 全空白输入会被归一为空——Kimpo 里**不存在**"内容是空字符串"这种状态 |
| 数值 | 没填 | 显式输入 `0` 是**有值**，不是空 |
| 日期/时间 | 没选 | 永远不会变成 1900-01-01 |
| 勾选（布尔） | **没有空** | 没勾 = 否。勾选框刻意不引入第三态 |
| 附件 | 没有上传任何文件 | — |
| 编号（发号器） | 尚未发号 | 发号后恒有值 |

## 5. 老用户快速适应指南

如果你从"没填自动变 0 / 空串 / 1900"的产品迁移过来，下面是习惯平移表。**绝大多数场景（第 1~3 行）你什么都不用改**：

| 你的老习惯 | 在 Kimpo 里 | 要改吗 |
|---|---|---|
| 直接写 `=数量 × 单价`，反正没填也是 0 | 行为完全一样（空在算术里当 0） | ❌ 不用改 |
| `=IF(金额 > 1000, …)` 之类的比较 | 行为一样（空数值在比较里当 0） | ❌ 不用改 |
| 拼接、LEN、文本函数随便用 | 行为一样且更好（空拼出来是空，不是 `"0"`） | ❌ 不用改 |
| 靠"空日期=1900"让日期区间筛选在没填时"不限"（如 `制单日期 >= 开始日期`，或 `datediff` 写法） | 行为保持不变且更可靠：**区间/模糊/"不等于"条件的参数没填 = 自动不限**（固定规则，无需任何设置）。都没填查全部、只填开始只限下界、只填截止只限上界 | ❌ 不用改 |
| 每条筛选后面手写 `或者 参数=''`（要"不限"）/ `并且 参数<>''`（要"没填不取"） | **原样可用**：`=''`/`<>''` 系统自动理解为"为空/不为空"（新模式里不存在空字符串，这个写法没有歧义），并且永远压过默认规则。也可以省掉样板：区间/模糊条件本来就"为空则不限"，等值条件本来就"为空则不取" | ❌ 一个字都不用改 |
| `列 形如 '%参数%'`、`charindex(参数,列)>0` 做模糊搜索，没填就查全部 | 改用「包含」条件，行为一致：参数没填 = 不限（默认开） | ✅ 换写法，行为不变 |
| 用 `=IF(日期 = "1900-1-1", "未填", …)` 判断日期没填 | 改成 `=IF(ISBLANK(日期), "未填", …)` | ✅ 改一处 |
| 用 `=IF(数量 = 0, "没填", …)` 判断没填 | 想清楚你到底问哪个：真问"等于零"就保持不动；问"没填"改用 `ISBLANK(数量)`——**老模式下这两个问题你被迫混为一谈，现在可以分开问了** | ✅ 按意图改 |
| 筛"数量 = 0"把没填的一起捞出来 | 捞不出没填的了。要没填的加一条"数量**为空**"（筛选器里现成的选项） | ✅ 改筛选条件 |
| 报表里手工剔除假 0 再算平均 | 不用剔了，平均数天然正确 | ❌ 删掉老补丁 |
| 给日期字段设一个"很早的默认值"占位 | 别设。真需要预填就配**字段默认值**（如默认今天）；不需要就留空 | ✅ 放弃占位习惯 |

三条心法，按优先级记：

1. **"像 Excel 一样"**——不确定空值会怎样时，想想 Excel 空单元格会怎样，八九不离十。
2. **想找"没填的"，用「为空」**——筛选器里的「为空/不为空」是为你新增的一等选项，不要再用 `= 0` 这种老暗号（`=''` 倒是被兼容收编了）。
3. **想问"没填吗"，用 `ISBLANK`**——公式里唯一需要新学的函数。

### 常见疑问 FAQ

**Q：用户根本不懂 null，会不会被吓到？**
不会，因为用户永远**看不到** null 这个词。没填的格子显示空白，筛选器里是中文的「为空」，公式里空值自动扮演 0/空文本。null 只是账本内部的记法，不是用户界面语言。

**Q：我就是想让没填的数量按 0 参与计算，怎么办？**
什么都不用做——公式里本来就按 0 算。如果你想让它**存**为 0（比如后续要导出给一个不认空值的老系统），给字段配默认值 0，让"0"成为用户看得见、改得了的真值，而不是系统偷偷垫的。

**Q：为什么"不等于 100"不包含没填的单据？**
因为"没填"意味着**不知道**它是多少——不知道的东西既不能说等于 100，也不能说不等于 100。这是数据库和 Excel 筛选共同的标准答案。想把没填的一起捞出来，加一条"或 金额为空"。

**Q：勾选框为什么没有"空"？**
一个勾选框在用户眼里只有勾和不勾两种样子，硬造出第三种"没碰过"状态只会让公式和筛选平添分支。所以 Kimpo 约定：没勾就是否。如果你的业务真需要"未表态"这一态，用单选字段（是/否/未表态）表达，别用勾选框。

**Q：我以前写 `制单日期 >= 主表.开始日期 并且 制单日期 <= 主表.截止日期` 做区间筛选，没填就查全部——现在还能这样吗？**
能，而且不用改写，也没有任何要你设置的开关——规则是固定的，跟你原来的直觉一致：**区间、模糊搜索、"不等于"这类条件，引用的参数（本报表的字段、系统变量、查询输入）没填时自动当没写**（都没填查全部，只填开始只限下界，只填截止只限上界）；**等值匹配（`单号 = 参数`）参数没填就是配不上**（就像以前空单号配不上任何单据一样）。这个判定很宽容——参数写在左边还是右边、正着写反着写、有没有套函数（如 `制单日期 >= 开始日期往前7天`、datediff 写法）都一样认得；"并且/或者"怎么连都正确（"或者"组里只当没写那一条，不影响另一条）。想反着来？**用你原来就会的写法**：要"没填也查全部"就补一句 `或者 参数为空`（老写法 `或者 参数=''` 原样可用）；要"没填就一条都不查"就补 `并且 参数不为空`（老写法 `<>''` 原样可用）——这两句永远压过默认规则。另外你用「空值默认」显式兜底（如 `IFNULL(开始日期, 2000-01-01)`）时，系统也绝不会替你裁。最后记住两种"空"的区别：`制单日期` 是**数据**，为空=这单没填（从严）；`开始日期` 是**参数**，为空=你没提要求。数量区间同理——最小数量填 `0` 是真要求（`≥0` 生效），不填才是不限，这个区分正是空值方案的红利。

**Q：筛选里"排序"和"分组"遇到空值会怎样？**
排序时空值**恒排在最后**（不管升序降序），不会一开屏先看到一堆空行；分组时没填的单据归成一个「(空白)」组，方便你一眼看到"还有多少没填"。

**Q：取数按"项目+分期"匹配，分期没填的行会互相匹配上吗？**
不会。匹配键任何一段为空的行**不参与匹配**（空 ≠ 空，和数据库 join 一个规矩）——否则所有没填的行会互相乱配，那是灾难。如果你的业务里"没填也算一档"，给那个字段配默认值（如"默认"），让它成为真值再参与匹配。

**Q：插件开发者要注意什么？**
读字段时空值是真实存在的状态（`TypedValue` 的 raw 为 null、读接口的 found 标志），写字段时"清空"就写 null——**永远不要**替用户垫 `""`/`0`/`1900-01-01`，那会污染账本并绕过整套空值语义。详见 aria/ 专题的读写契约。
