## FACTS（事实，编号可引用）

### 基础语义

- **F1** 每个 date/datetime 字段有布尔属性 **auto_timezone**（数据表管理器"自动匹配时区"，仅 date/datetime 可勾，**默认 false 且三处保证**：迁移列 DEFAULT 0 / 创建接口零值 / UI 默认不勾）。它声明的是**该列的存储语义**，不是显示偏好。**naive 是常态，auto 是显式选择**；谁勾选，谁对该字段的高级用法（F14/F15）负责。
- **F2** `auto_timezone=false`（墙钟/naive，默认）：字面即语义。DB 列、业务层、屏幕三层**字面全等**；任何用户、任何时区看到同一串数字；全链零换算。
- **F3** `auto_timezone=true`（绝对时刻）：DB 列存 **UTC 墙钟字面值**；DB 之外的一切层（API 响应、Mirror(Aria)、编辑器、公式、前端、插件可见值、取数直引结果）一律是**当前会话用户的本地墙钟**。示例：上海用户（UTC+8）录入 `21:00` → DB `13:00:00` → 上海用户读回 `21:00`，纽约用户（7 月 UTC−4）读回 `09:00`。
- **F4** UTC↔本地换算**只发生在宿主的边界**（读写库边界 + 取数编译/读出口，均为宿主实现），按会话用户的时区偏好（IANA 名，如 `Asia/Shanghai`）执行。插件、编辑器、前端**均无换算责任，也无换算权利**。
- **F5** 插件经 Mirror 读到的日期值 = 会话用户本地墙钟（与用户格子所见一致）；写入 Mirror / 经 `Host.Record()` 批量写库，也必须给本地墙钟。插件视角下 auto_timezone 字段与 naive 字段**处理方式完全相同**。
- **F6** 业务日期值的规范文本形态 = 空格分隔串：datetime → `yyyy-MM-dd HH:mm:ss`，date → `yyyy-MM-dd`。RFC3339/ISO 形态（含 `T`/`Z`/偏移）是内部泄漏，不得产出；若读到，按"取墙钟面、忽略 Z 与偏移"解析。
- **F7** 系统审计列（created_at / updated_at 等）恒为绝对时刻：DB 存 UTC，展示层（含记录列表）按用户时区偏好换算。与 auto_timezone 机制平行，无字段开关。
- **F8** DB 一律使用**无时区列类型**且 date 与 datetime **物理同型**（MySQL DATETIME(3) / PG TIMESTAMPTZ 之外的族不启用连接时区换算；禁用 MySQL TIMESTAMP 等会话敏感型）。字面比较/排序/分组三家数据库行为一致；**禁止依赖会话时区**（`SET time_zone` 等连接级状态被架构层否决）。
- **F9** 匿名会话 / 用户未设时区偏好 → 换算降级为 no-op（naive 直通），不报错。因此"读到的值恰好等于 DB 字面值"在匿名场景是预期行为。
- **F10** 翻转 auto_timezone 开关**不迁移存量数据**：既有字面值的解释语义整体改变。生成建模建议时：字段创建时定死，勿建议对有数据的字段翻转。
- **F11** **date 值面按 datetime 处理**（2026-07-13 拍板）：date 值 = `00:00:00` 墙钟，与 datetime 同规则折算，物理列同为 DATETIME(3)，auto+date 读写可无损往返。语义提醒：勾 auto 的 date 字段是"指向物理时刻锚点的日期"（各时区看到的日期**可以不同**，如全球活动日）；生日/合同日期类纯墙钟日期**勿勾**（默认不勾即正确）。

### 取数链路（2026-07-13 起已实施换算）

- **F12** 取数（统一表达式模型通道，含动作取数/候选查询/variants 命中/条件格式）的时区换算**已实施**，宿主自动完成两侧：①**参数侧**——与 auto 列比较的绑定值（本报表字段值/系统变量/常量，含 IN 成员、Excel DateSerial 浮点）编译期折成 UTC 再绑定；②**结果侧**——auto 列**裸直引**输出（`o_*` 目标列与 `m_*` 匹配键列）读出口自动转会话用户本地墙钟。插件拿到的直引列值 = 用户本地墙钟，直接用。
- **F13** 结果侧换算**只认裸列直引**：函数/聚合/raw 表达式包裹的输出**不换算**（保守规则，无法静态断定其时区语义）。消费这类列时，值是 UTC 域的计算结果。
- **F14** **SQL 内日期成分提取不做平移（v1.1 拍板，设计者责任）**：`date_diff`、raw 表达式里的 `YEAR()/DATE()/按天分组` 作用在 auto 列上时按 **UTC 字面**计算——UTC 日界 ≠ 用户日界（东八区本地 00:00–08:00 的数据落在 UTC 前一日）。平台不在 SQL 里插入 CONVERT_TZ；需要用户日界时，由调用方在应用侧算好**字面边界值**注入条件（见 M4）。
- **F15** **视图零时区机制（v1.1 拍板，设计者责任）**：auto 字段进入视图后，视图输出列一律按 **naive** 对待——参数不折、结果不转、元数据不传染。生成视图建模时**建议只使用 naive 字段**；确需 auto 字段时，消费方按 UTC 字面处理并自担口径。
- **F16 已知遗留**：旧文本 KimpoSQL 通道（动作配置的 `params.kimpo_sql` 老形态）零换算——参数与结果均为 UTC 字面。新表达式模型通道（F12）无此问题；遇老形态配置建议迁移。

### 时间的产生

- **F17** 系统变量按**会话用户时区**求值：字段默认值 `current_date`/`current_datetime`（预填+补填）、动作系统 `current_time`/`today`。`current_time` 输出空格墙钟串（曾为 RFC3339，2026-07-13 起统一）。匿名/无偏好回落服务器本地。
- **F18** 自动编号的 date 段按**服务器时钟**（2026-07-13 拍板）：流水序列必须唯一口径，不随用户时区变。

## VALUE-FORMAT（值形态规范）

```
写入（Mirror WriteField/WriteFields 的 TypedValue.raw / text，及 Host.Record() 批量写）:
  datetime: "2026-07-11 21:00:00"   ← 用户此刻应看到的本地墙钟，空格分隔全精度
  date:     "2026-07-11"            ← 值面等价 "2026-07-11 00:00:00"（F11）
禁止产出: "2026-07-11T21:00:00Z" / "2026-07-11T21:00:00+08:00" / Unix 时间戳当日期串
读取时兼容: 空格串（正常）；若遇 RFC3339 → 取墙钟面忽略 Z/偏移；数值型 Excel 日期序列号可能出现于编辑器内部值，插件不应依赖
```

## DECISION TABLE（时区决策表）

| 场景 | 决策 |
|---|---|
| 插件读到日期值（Mirror / 取数直引列），要不要转时区？ | **不要**（F4/F5/F12）。它已是会话用户本地墙钟，直接用。 |
| 插件写日期值，要不要先转 UTC？ | **不要**（F4/F5）。给本地墙钟；转了就是双转事故（用户选 21:00 显示成 04:00 的历史故障根因）。 |
| 取数条件里 auto 列与表单值/常量比较 | 直接写，宿主编译期折 UTC（F12）——不要自己预折。 |
| 取数结果里函数/聚合/raw 包裹的列 | UTC 域计算结果（F13），勿当本地墙钟展示。 |
| 需要"当前时间"填给用户 | 用系统变量（F17）或会话用户语境的墙钟；MUST NOT 拿服务器本地时钟冒充用户墙钟。 |
| 手写 SQL 给 UTC 语义列（auto 字段/系统列）造数 | 用 `UTC_TIMESTAMP()`；MUST NOT 用 `NOW()`（服务器时区墙钟）。 |
| SQL 里比较 auto 字段与 naive 字段 | **禁止**直接互比（UTC vs 用户墙钟，无业务意义）。 |
| 对 auto 字段 `GROUP BY DATE(col)` / `date_diff` / raw 内 `YEAR()` | 按 **UTC 日界**计算（F14），平台不平移。需用户日界 → 应用侧算好字面边界值注入（M4）。 |
| 视图里用 auto 字段 | 平台零换算（F15）。建议改用 naive 字段；坚持用则按 UTC 字面自担口径。 |
| 老文本 KimpoSQL 取数（params.kimpo_sql） | 零换算遗留（F16），值按 UTC 字面处理并建议迁移到表达式模型。 |
| 查库值与页面显示差 = 用户时区偏移 | 预期行为（F3/F7），不是 bug；勿生成"修复"代码。 |
| naive 字段查库值 ≠ 页面显示 | 真 bug，应上报，而非在插件里打补丁换算。 |

## MUST

- M1 处理任何业务日期值时，一律按"会话用户本地墙钟"语义编码（F3/F5/F12），与 naive 字段同构处理。
- M2 写入用空格分隔规范串（F6）；date 档只给日期部分。
- M3 解析外来日期串时容忍 RFC3339 形态但**只取墙钟面**（忽略 Z/偏移）。
- M4 涉及"按日/按周"相对条件或按用户日界统计时，由调用方在应用侧算好**字面边界值**注入查询条件（如上海用户的"今天" = `[前一日 16:00:00, 当日 16:00:00)` 的 UTC 界），不依赖 SQL 端 `NOW()`/`CURDATE()`，也不指望平台在 SQL 里平移（F14）。
- M5 给用户展示系统审计时间（created_at 等）时按用户时区偏好格式化（F7）。
- M6 生成模板/字段建模建议时：默认不勾 auto_timezone（F1）；仅对"跨时区必须同时发生"的时刻字段建议开启，并同步告知 F14/F15 的设计者责任。

## MUST NOT

- N1 MUST NOT 在插件/前端/模板自动化中执行任何 UTC↔本地换算（F12 实施后无任何例外；唯一合法的"手工对齐"是 M4 的边界值预计算与 F13/F15/F16 场景的显式 UTC 口径处理，须注释标明依据的 FACT 编号）。
- N2 MUST NOT 产出含 `T`/`Z`/时区偏移的日期串写入业务字段。
- N3 MUST NOT 用服务器本地时间冒充用户墙钟；MUST NOT 用 `NOW()` 给 UTC 语义列造数。
- N4 MUST NOT 在 SQL 中让 auto_timezone 字段与 naive 字段直接比较；MUST NOT 在视图里混合 auto/naive 来源到同一输出列。
- N5 MUST NOT 建议对已有数据的字段翻转 auto_timezone 开关（F10）；MUST NOT 对生日/单据日期类纯墙钟日期建议开启（F11）。
- N6 MUST NOT 把"查库字面值 ≠ 页面显示值"当 bug 修（F3/F7 预期）；也 MUST NOT 反向"修复"——绝对时刻字段查库若与页面**相等**才是写入换算失效的征兆。
- N7 MUST NOT 依赖数据库会话时区（`SET time_zone` / session_timezone 等，F8）；MUST NOT 建议把列改为 TIMESTAMP/timestamptz 等连接敏感类型。

## TYPICAL SEQUENCE（动作插件写一个 auto_timezone 字段）

```go
// 目标：把"会议开始"设为用户视角的 2026-07-11 21:00（上海用户）
// 正确：直接给本地墙钟，宿主在 DB 边界折算为 13:00 UTC 落库
tv, _ := json.Marshal(map[string]any{"raw": "2026-07-11 21:00:00", "text": "2026-07-11 21:00:00"})
_, err := mirror.WriteField(ctx, sessionID, tableID, "main:0", fieldID, tv, writeGrant)
// 错误示范（双转事故）：先把 21:00 自行转成 "2026-07-11 13:00:00" 再写 —— 用户会看到 05:00
```
