# 日期时间与时区：数据库里存的，为什么和屏幕上看到的不一样 —— 面向人类的讲解

> 读者：模板设计者（决定字段要不要勾"自动匹配时区"）、插件开发者（读写日期字段）、以及**直接查数据库排障的人**。
> 读完你会明白：同一个时间字段，数据库字面值和用户屏幕显示什么时候相等、什么时候不等、差多少、为什么；插件拿到的又是哪一个；以及三个"平台故意不管"的特殊点。
> 配套：给 AI 编程助手的结构化版本在同目录 `timezone-for-llm-agents.md`；空值语义见 `empty-values-for-human-developers.md`；插件读写通道见 `../mirror/`，查询见 `../expression/`。
> 修订：2026-07-14——取数链路换算已实施；date 按 datetime 处理；新增 §6「平台的责任边界」（三个特殊点）。

## 1. 先分清两种"时间"：它们根本不是一回事

**"合同签订日期是 2026-07-11。"** 这句话全世界任何人来读都是 2026-07-11——纽约的律师不会把它读成 7 月 10 日。这个日期**字面就是信息的全部**，它不指向宇宙中某个物理时刻。我们叫它**墙钟时间**。生日、入职日期、发票日期、报销单上的消费日期，全是这一类。

**"全球发布会 2026-07-11 21:00 开始。"** 这句话缺了半截——21:00 是哪里的 21:00？如果是上海的 21:00，纽约同事必须在**他的**早上 09:00 上线。这种时间指向**宇宙中唯一的一个物理时刻**：各地钟表读数不同，时刻相同。我们叫它**绝对时刻**。跨时区的会议开始时间、打卡时间、直播开始时间，是这一类。

**判断口诀：换个时区的人来看，数字该不该变？**

- 不该变（生日永远是 5 月 1 日）→ 墙钟时间。
- 该变（上海 21:00 的会，纽约人看到 09:00 才对）→ 绝对时刻。

绝大多数业务日期是墙钟时间。所以 Kimpo 的默认就是墙钟：**字面进、字面出、永不换算**。绝对时刻是显式开启的例外。

## 2. 一个字段级开关：「自动匹配时区」（默认关闭）

在数据表管理器里，每个 **date / datetime** 字段有一个布尔属性「**自动匹配时区**」（其它类型字段勾不了，**默认不勾**——数据库列缺省、创建接口缺省、界面缺省三处一致）：

| | 不勾（默认，墙钟） | 勾上（绝对时刻） |
|---|---|---|
| 语义 | 字面就是全部，所有人看到同一串数字 | 指向唯一物理时刻，各时区看到各自的墙钟 |
| 数据库列里存什么 | **用户填的字面值原样** | **UTC 墙钟**（上海用户填 21:00 → 存 13:00） |
| 上海用户看到 | 21:00 | 21:00 |
| 纽约用户看到 | 21:00（一样） | **09:00**（不一样） |
| 该用哪个 | 生日、合同日期、消费日期……想不清楚就用它 | 跨时区协作里"必须同时发生"的时刻 |

勾上之后，平台向用户兑现三条承诺：

1. **选什么就是什么**——选择器里选 21:00，格子里就显示 21:00；
2. **所见即所算**——公式、画布、导出、取数条件用的都是你看到的 21:00，不是暗地里的什么值；
3. **切时区，时间跟着变**——把个人时区偏好从上海改成纽约，同一条记录从 21:00 变成 09:00。物理时刻没变，变的是你的钟。

要兑现第 3 条，系统就必须记住那个物理时刻——**这就是数据库必须存 UTC 的原因**。它不是"另一套逻辑"，而是第 3 条承诺的必然结果。

三个注意：

- **date（纯日期）字段与 datetime 同规则处理**：date 值在系统里等价于"当天 00:00:00 的墙钟"，物理列与 datetime 同型（能承载换算出的时间分量），勾了自动匹配时区后读写可以无损往返。但语义上要想清楚：勾了 auto 的日期是"指向某个物理时刻锚点的日期"——上海的 7 月 12 日在纽约用户眼里**就是 7 月 11 日**，这是正确行为而非 bug。生日、单据日期这类纯墙钟日期**不要勾**（默认不勾即正确）。
- **开关不迁移存量数据**。字段里已经有数据后再翻转开关，旧字面值的解释语义会整体改变（原来当墙钟的 21:00 突然被当成 UTC）。原则：**字段建好时就定下来，别对有数据的字段翻开关**。
- **勾选即责任**：勾 auto 的设计者需要了解 §6 的三个特殊点（日期成分统计、视图、老取数通道），那些场景平台故意不插手。

## 3. 重点：数据库里的值 ≠ 屏幕上的值（差异到底在哪一层）

一个时间值从落库到进入眼睛，穿过三层：

```
┌─────────────┐      ┌────────────────────┐      ┌─────────────┐
│ 数据库列字面值 │ ───▶ │ 业务层（接口/内存/插件） │ ───▶ │  屏幕显示值   │
└─────────────┘      └────────────────────┘      └─────────────┘
```

用一条真实记录对照（"会议开始"勾了自动匹配时区；用户 A 偏好上海 UTC+8，用户 B 偏好纽约，7 月为 UTC−4）：

| 层 | 墙钟字段「合同日期 21:00」 | 绝对时刻字段「会议开始」 |
|---|---|---|
| **数据库列字面值**（`SELECT` 直接看） | `2026-07-11 21:00:00` | `2026-07-11 13:00:00` ← **UTC，全链唯一与显示不同的一层** |
| 业务层给用户 A 的值 | `2026-07-11 21:00:00` | `2026-07-11 21:00:00`（13:00 + 8h） |
| 用户 A 屏幕 | 2026-7-11 21:00:00 | 2026-7-11 21:00:00 |
| 业务层给用户 B 的值 | `2026-07-11 21:00:00`（一样） | `2026-07-11 09:00:00`（13:00 − 4h） |
| 用户 B 屏幕 | 2026-7-11 21:00:00 | 2026-7-11 09:00:00 |

这张表读出三条铁律，请背下来：

**① 墙钟字段三层全等。** 数据库看到什么，用户就看到什么，谁看都一样。核对墙钟数据时，查库所见即所得；如果查库和页面不一致，那是 bug，报修。

**② 绝对时刻字段只有数据库那一层是 UTC。** 换算只发生在平台边界上（读写库边界、取数出入口）；跨过边界之后，整个系统里流动的都是"**当前会话用户的本地墙钟**"。所以：
- 页面显示 21:00、查库看到 13:00，**不是 bug**——差值恰好等于该用户的时区偏移；
- 反过来，如果查库也看到 21:00，说明写入换算**坏了**，这才要报修。

**③ 系统时间（created_at / updated_at 等审计列）永远按绝对时刻处理。** 数据库统一存 UTC，页面（含记录列表）按用户时区偏好显示。查库看到创建时间比页面早 8 小时（上海用户），同样正常。

> **一句话记住**：*换算只住在平台边界；边界之内（数据库列）绝对时刻是 UTC，边界之外（其它一切地方）永远是"这个用户自己的钟"。*

## 4. 取数与条件：平台已经替你换算好的部分

取数（动作取数、列表控件候选查询、条件化配置命中、条件格式）走统一表达式模型时，平台在两头自动换算：

- **条件参数**：条件里"绝对时刻字段 = 本报表某字段值 / 常量 / 系统变量"——你给的是本地墙钟，平台在编译 SQL 时自动折成 UTC 再比。**"发生时间 = 填写时间"这类条件在任何时区都正确命中**，不需要你预先转换（转了反而双转出错）。
- **结果值**：取数结果里**直接引用**绝对时刻字段的列（包括按行列匹配的匹配键列），平台读出时自动转回你的本地墙钟——和格子里看到的一致。

**但有保守边界**：被函数/聚合/自由文本表达式**包裹**的输出（如 `max(发生时间)` 之外的复杂表达式、raw 里算出来的值）不做自动换算——平台无法静态判断包裹后的时区语义，宁可不动。这类值按 UTC 口径理解。

**系统时间的产生**也已按用户时区：字段默认值的 `current_date`/`current_datetime`、动作系统变量 `current_time`/`today`，求出来的都是**会话用户的墙钟**（匿名或未设偏好时回落服务器时间）。唯一例外是**自动编号里的日期段——固定按服务器时钟**：流水号的口径必须全局唯一，不能上海用户生成的编号和纽约用户不同段。

## 5. 直接查数据库的人须知（运维 / 排障 / 写 SQL）

- **核对绝对时刻值是否正确**：`页面显示的墙钟 − 用户时区偏移 = 数据库字面值`。上海用户显示 21:00，库里就该是 13:00。
- **手工 UPDATE 绝对时刻字段**：你写进去的字面值会被系统当作 **UTC** 解释。想让上海用户看到 21:00，就写 `13:00:00`。
- **手工造数别用 `NOW()`**：MySQL 的 `NOW()` 返回服务器时区的墙钟。给"存 UTC"的列（绝对时刻字段、created_at 等系统列）造数，用 **`UTC_TIMESTAMP()`**。
- **date 字段的列型是 DATETIME(3)**（不是 DATE）：date 值以 `yyyy-MM-dd 00:00:00`（naive）或折算后的 UTC 时刻（auto）形态落列，查询显示层只呈现日期部分。
- **两类字段别在 SQL 里直接互比**：`WHERE 会议开始 > 合同日期` 是拿 UTC 和用户墙钟比大小，结果没有业务意义。

## 6. 平台的责任边界：三个"故意不管"的特殊点 ⚠️

平台保证三条底线：**存得对**（UTC 落库）、**比得对**（条件参数折算）、**显示得对**（读出口换算）。以下三处平台**有意不插手**，属于勾选 auto 的设计者责任——这是产品决策（2026-07-13 拍板），不是缺陷：

**特殊点一：SQL 里取"日期成分"按 UTC 算。**
对绝对时刻字段做 `按天分组`、`date_diff`、自由文本表达式里的 `YEAR()/DATE()`——这些运算作用在数据库字面值（UTC）上，分的是 **UTC 的日界**。上海本地 00:00–08:00 产生的数据会落进 UTC 的前一天。平台不在 SQL 里自动平移（保持查询引擎简单、行为可预测）。需要按用户日界统计时，由设计者在条件里给出**算好的边界值**（例如上海用户的"今天" = UTC 的 `[昨日16:00, 今日16:00)`），或者干脆用 naive 字段做统计口径。

**特殊点二：视图里的绝对时刻字段，平台一律不换算。**
视图是全用户共享的数据库对象，没有"当前用户"可言。绝对时刻字段进入视图后按 naive 对待：条件参数不折算、结果不换算。**建议视图尽量只用 naive 字段**；确要用绝对时刻字段时，视图消费方要清楚拿到的是 UTC 字面值，口径自理。

**特殊点三：老文本取数通道零换算。**
动作配置里的旧形态裸 KimpoSQL 取数（非表达式模型）不做任何时区换算，参数与结果均为 UTC 字面。新配置一律走表达式模型（§4 的自动换算全部生效）；遇到老配置建议迁移。

为什么这样划界？因为**默认不勾 auto 意味着 naive 是常态**——绝大多数字段字面进出，按天分组天然正确，根本没有时区问题。auto 是少数跨时区场景的显式选择，为它把查询引擎、视图体系全部复杂化不划算；平台把底线守牢，口径判断留给最了解业务的人。

## 7. 排障速查表

| 现象 | 判断 |
|---|---|
| 查库时间比页面早 8 小时 | 正常。绝对时刻字段或系统列存 UTC，页面按你的时区（上海 +8）显示。 |
| 墙钟字段查库和页面不一致 | **异常**。墙钟三层应全等，报 bug。 |
| 换了时区偏好，历史数据"变了" | 正常且正确。绝对时刻字段显示随观察者的钟变，物理时刻没变。墙钟字段若变了才是 bug。 |
| 两个用户对同一字段各执一词（21:00 vs 09:00） | 都对。这正是绝对时刻的正确形态。 |
| 勾了 auto 的日期字段，纽约用户看到的日期比上海早一天 | 正常（§2）。auto 日期指向物理时刻锚点，各时区日期可不同。 |
| 取数条件"发生时间=填写时间"跨时区命不中 | **异常**（§4 已自动折算），报 bug——除非走的是老文本通道（§6 特殊点三）。 |
| 报表按日汇总，凌晨单子跑进前一天 | 绝对时刻字段按 UTC 日界分组所致（§6 特殊点一），设计者责任范围，预期行为。 |
| 视图里的绝对时刻字段显示 UTC | 预期（§6 特殊点二），视图零换算。 |
| 页面出现 `2026-07-11T13:00:00Z` 这种带 T 带 Z 的原始串 | **异常**。某条链路把内部形态漏出来了，报 bug。 |
| 翻转"自动匹配时区"开关后旧数据乱了 | 已知边界（§2）：开关不迁移存量，语义解释即时改变。建好就定，别翻。 |

## 8. FAQ

**Q：我怎么知道该不该勾"自动匹配时区"？**
A：用 §1 的口诀问自己："换个时区的人来看这个时间，数字该不该变？"想不清楚就不勾——墙钟是安全默认（也是产品默认），勾错了造成的语义混乱比不勾大得多。勾之前再看一眼 §6 的三个特殊点。

**Q：为什么不干脆全部存用户本地时间？**
A：绝对时刻字段是跨用户共享的——上海存 21:00、纽约存 09:00 指同一时刻，库里必须有一个**唯一规范写法**才能比较、排序、去重。全世界的惯例就是 UTC。业界也翻过这个车：Mixpanel 早年按项目时区换算后落盘，用户改时区历史数据全错且不可回溯，2023 年整体推倒改成了"存 UTC、查询期换算"。

**Q：为什么不全部存 UTC（连墙钟也是）？**
A：墙钟根本没有"对应哪个物理时刻"这回事。生日 5 月 1 日转成 UTC 是几点？这个问题本身无意义。硬转的系统都会在"差一天"上翻车。

**Q：数据库为什么不用带时区的列类型（如 `TIMESTAMP`/`timestamptz`），或者查询时 `SET time_zone` 让数据库自己换算？**
A：带时区列类型随连接时区自动换算，三家数据库行为不一、查询结果取决于谁来连；会话时区在共享连接池下会把状态泄漏给下一个请求（业界如 ClickHouse 的同类功能至今挂着 Beta 警告）。Kimpo 一律用无时区类型 + 显式字段语义，换算收敛在平台边界一处做——这也是 Rails/Django/Salesforce 等成熟系统的共同选择。

**Q：为什么平台不把按天分组也自动换算了（§6 特殊点一）？**
A：能做但不值得。默认关闭意味着绝大多数字段是墙钟、天然没这个问题；为少数 auto 字段把查询引擎塞进自动平移（还要求数据库灌时区表、处理夏令时），复杂度全员买单。平台守住"存/比/显"三条底线，日界口径这种业务判断交给勾选它的设计者。
