# 流程引擎使用指南 —— 面向普通用户与流程设计者

> 读者：要在 Kimpo 里**画审批流、办理审批**的业务用户与管理员。不需要任何编程知识。
> 读完你会明白：一条审批流从画出来、发布、绑定到表单，到别人提交单据、你在任务中心办理，中间每一步发生了什么；每种节点是干什么的；每个设置项背后是什么意思。
> 配套：给开发者看的原理版在同目录 [for-human-developers.md](for-human-developers.md)，给 AI 编程助手看的结构化版在 [for-llm-agents.md](for-llm-agents.md)。

---

## 1. 流程是什么：给表单装上"审批传送带"

Kimpo 的模板（表单）本身只管"填数据、存数据"。**流程**给它加上一条传送带：

> 有人提交一张单据 → 单据被锁住、进入审批 → 按你画好的路线图，一站一站送到该审批的人面前 → 每个人同意/不同意 → 全部走完，单据盖章定局（通过或不通过）。

三个核心概念：

| 概念 | 是什么 | 类比 |
|---|---|---|
| **流程定义** | 你在设计器里画的那张"路线图"（节点 + 连线 + 每个节点的设置） | 一张审批路线图纸 |
| **流程绑定** | 把某条流程挂到某个模板上："这个表单提交时走这条流程" | 把图纸贴在某个窗口上 |
| **流程实例** | 某一张具体单据按图纸跑起来的那一次审批。一张单据提交一次 = 一个实例 | 一件正在传送带上的包裹 |

**单据的三种流程状态**（决定单据还能不能改）：

- **未进流程**：随便改，正常的草稿/编辑状态。
- **审批中（processing）**：单据整体锁定只读——除非某个审批节点明确"放行"了几个字段给当前办理人改。
- **已通过（approved）/ 已拒绝（rejected）**：定局，单据锁定，可在列表里按状态筛选。

被拒绝或被撤回的单据会回到可编辑状态（撤回回到"未进流程"），修改后可以再次提交。

## 2. 五分钟上手：画一条最简单的请假审批流

以"员工提交请假单 → 直属主管审批 → 通过"为例，完整走一遍。

**第 1 步：新建流程。** 打开你的应用，在左侧应用树上新建一个流程（与新建模板同级的入口），起名"请假审批"。进入流程设计器——一块类似 Visio 的画布。

**第 2 步：画路线。** 画布初始有一个**开始**节点。从左侧形状库把**审批**节点拖到画布上，从"开始"拉一条线连到"审批"。（不用再画"结束"——没有后续节点的末端，发布时会自动补上结束，见 §8.3。）

**第 3 步：配置审批节点。** 点中审批节点，右侧属性面板里设置：

- **审批人**：添加一条规则，选"直属主管"。含义：谁提交的单，就送到谁的直属上级面前。
- **按钮**：默认已带"同意"和"不同意"两个按钮，不同意默认动作是"退回发起人"。先不动它。

**第 4 步：发布。** 点"发布"。设计器会先做体检（发布校验），比如审批节点没配审批人会被拦下并定位到具体节点。校验通过后生成 **1.0 版本**——发布后的版本永远不可变（见 §9）。

**第 5 步：绑定到表单。** 把这条流程绑定到"请假单"模板。绑定后，该模板的单据提交时就会走这条流程（表单上会出现"提交"入口）。

**第 6 步：体验一次审批。** 用普通账号填一张请假单并提交 → 单据锁定、变为"审批中"→ 主管在左侧菜单**任务中心 → 待办**里看到这张单 → 点开办理面板：左边是只读的单据内容，右边是流程进度图，底部是"同意/不同意"按钮和意见框 → 点"同意"→ 流程走到终点，单据变为"已通过"。

这就是全部骨架。剩下的一切——多人会签、条件分支、超时催办、退回重提——都是在这个骨架上加配置。

## 3. 设计器操作手册

流程设计器对标 Visio 的操作手感：

| 操作 | 怎么做 |
|---|---|
| 建节点 | 从左侧**形状库拖拽**到画布（这是唯一的建节点方式，防误触） |
| 连线 | 从节点边缘的连接桩拉线到目标节点；拉到目标附近会自动吸附到最近的连接桩，连线末段自动垂直入边，图面整齐 |
| 改线 | **点中一条线，头尾会出现拖拽点**，拖动端点即可改接到别的节点 |
| 条件/默认线 | 点中连线，在属性面板配置条件表达式，或勾选"默认分支"（见 §8） |
| 移动画布 | 按住 **Ctrl 拖动**空白处平移；滚轮缩放；右下角缩略图导航（最大 100% 显示） |
| 对齐 | 拖动节点时有对齐参考线自动吸附；按住 **Alt** 可临时禁用一切吸附，精细摆放 |
| 撤销/重做 | Ctrl+Z / Ctrl+Shift+Z（或 Ctrl+Y） |
| 删除 | 选中节点/连线按 Delete；删除节点会连带删除相连的线 |
| 校验定位 | 发布时校验不通过，错误列表可点击，直接跳转定位到出问题的节点/连线 |

**保存与发布是两回事**：随时保存的是**草稿**（只有你在改，别人看不到）；点**发布**才生成新版本、对提交生效。

## 4. 节点全集：12 类节点每个是干什么的

节点分三族：**人任务**（要人来处理）、**结构**（控制路线走向）、**自动**（系统自己执行）。

### 4.1 开始（start）

每条流程有且只有一个。它是流程的入口，同时承担三个身份：

- 填报人点"提交"的落点——提交成功即从这里出发；
- 进度图的第一环（记录发起人、发起时间）；
- "退回发起人"的落脚点、撤回窗口的锚。

无需配置。

### 4.2 审批（approval）—— 最常用的节点

"要有人**表态**"的节点：同意放行，不同意按你配的动作收场。设置项最多，全部展开在 §5～§7 讲。核心设置一览：

| 设置项 | 管什么 |
|---|---|
| 审批人 | 这一站送到谁面前（十种指定方式，§5） |
| 多人模式 | 送到多个人面前时怎么算过（会签/或签/依次/比例签，§6） |
| 按钮 | 办理人能点什么："同意"固定向下走；"不同意"必须配后果（整单终止 / 退回上一步 / 退回发起人） |
| 审批意见 | 是否显示意见框、是否必填、标题文案 |
| 字段权限 | 本节点办理人可**编辑**哪些字段、**隐藏**哪些字段（没列的一律只读） |
| 摘要 | 任务列表里显示的一句话摘要（如"张三的请假单：3 天"），支持取单据字段拼装 |
| 超时 | 超过多久提醒、要不要升级处理（§7） |
| 自动办理 | 满足某个条件自动通过本站（如"金额 < 1000 跳过本级"） |
| 退回设置 | 退回时要求清空重填哪些字段；重提后直达本节点还是重走全程（§10.3） |

### 4.3 办理（handle）

"办事不表态"的节点：按钮语义是**完成/提交**，没有"不同意"的概念。典型场景：接力填单——A 填一半提交，流程送到 B，B 补填自己负责的字段再提交。

独有能力：**补填必填**——在字段权限里指定"本环节必须填完的字段"，不填完提交不了。其余设置（办理人指定、超时、摘要等）与审批节点完全相同。

### 4.4 抄送（cc）

"知会，不挡路"：流程走到这里，给被抄送人发一条抄送任务后**立即继续向下走**，不等任何人。被抄送人在任务中心"抄送"页签里看到，点"已阅"归档即可，支持批量已阅。

注意：抄送不能当结束节点用——把抄送放在流程末端是可以的（末端抄送发完，流程自然结束），但"结束"的判定不靠它（见 §8.3）。

### 4.5 终止（terminate）

"提前判死刑"的显式出口：条件线可以直接连到终止节点，到达即**整单终止**——实例作废、单据变"已拒绝"、所有还没办完的任务统统撤销。典型用法：某个分支条件下（如"金额超限且无预算"）直接毙掉，不用一层层退。

### 4.6 并行网关（parallel_split / parallel_join，一对）

- **并行-分**：一个节点之后同时展开多条支线，各支线**同时**推进（如：合同审批同时送法务和财务）。分叉出去的线不需要配条件——它们全都会走。
- **并行-合**：等待**全部支线都到齐**才继续向下。少一条都不走。

配套规矩：并行期间**禁止跨支线退回**（法务支线上的人不能把单退到财务支线里去），设计器和引擎都会拦。"退回发起人"不受限（它回到分叉之前）。

### 4.7 动作（action）

流程里的"机器人工位"：到达后自动调用平台里的一个自动化能力（由动作插件提供，如"更新库存""生成台账记录"），执行完继续向下。设置项：

- **能力**：调用哪个能力 + 参数怎么取（常量 / 单据字段 / 发起人 / 流程变量）；
- **失败策略**：失败后重试几次；重试耗尽后走"错误分支"（一条标记为错误出口的连线）还是"转人工"（生成一个人工任务，交管理员确认后继续）；
- **出参回填**：执行结果可以存进"流程变量"，供后面的条件分支或其他节点参数使用（按需开启）。

### 4.8 子流程（subprocess）

把另一条流程当作一个节点嵌进来：到达后**发起那条子流程并等它办完**，子流程通过了父流程才继续；子流程被终止/撤回，父流程也整单终止。支持传参进出（父流程变量 ↔ 子流程变量），进度图上可以下钻查看子流程内部走到哪了。子流程用自己的最新发布版本，禁止循环嵌套（A 套 B、B 又套 A 会被拒绝）。

### 4.9 分发填报（dispatch）

"一单变多单，各填各的"：按名单（复用审批人的十种指定方式圈人）给**每个人生成一张独立的子单据**（可预填初始字段值），每人填自己那张，**全部人都交齐**后主流程才继续。典型场景：部门预算填报——发给 10 个部门负责人，每人填自己部门的，全交齐后进入汇总审批。

### 4.10 人工定路（manual_route）

"下一步去哪，办理人现场定"：办理人提交时**手动选择**下一个节点和下一步的办理人。设计者可以圈定允许选择的节点范围（不圈=当前节点所有出边都可选）。适合无法预先画死路线的场景（如疑难工单的转派）。

### 4.11 延时（delay）

"到点再走"：到达后等待指定时长（如 48 小时）再继续。典型场景：公示期——审批通过后公示两天，无异议自动进入生效步骤。等待由平台定时器驱动，不占任何人的待办。

### 4.12 结束（end）

流程的终点。到达即定局四件套：实例状态变"已通过"、单据变"已通过"并锁定、记录终审人/终审时间、通知发起人。**你可以不画它**——发布时系统会给没有后续节点的末端自动补上（见 §8.3）。

## 5. 审批人怎么指定：十种取人规则

在审批/办理/抄送/分发节点的"参与者"设置里添加规则。**可以加多条规则，结果取并集**（都算数），系统自动去掉重复的人。

| 规则 | 含义 | 例子 |
|---|---|---|
| **指定成员** | 直接点名若干人 | "审批人：王财务、李行政" |
| **角色** | 某个角色下的全部成员；可叠加部门关系过滤（只要和发起人同部门的 / 上级部门的…） | "财务角色 + 与发起人同部门" |
| **直属主管** | 发起人的第 N 级主管 | N=1 是直属上级，N=2 是上级的上级 |
| **部门主管** | 发起人主部门的主管；或"单据里某个部门字段"所指部门的主管 | 报销单上填了"费用归属部门"，就送那个部门的主管 |
| **发起人** | 发起人自己 | 常用于"发起人确认"环节 |
| **表单人员字段** | 单据里某个"人员"字段选中的人 | 单据上有"指定审核人"字段，选谁就送谁 |
| **连续多级主管** | 从直属主管一路向上审，直到满足终止条件（审到第 N 级 / 遇到某角色的人为止）；中间某级空缺自动跳过 | "逐级审到总监为止" |
| **上一步办理人** | 上一个节点实际办理的人 | "谁初审的，复核阶段再确认一次" |
| **发起人自选** | 提交时由发起人自己挑（可限定范围、单选/多选） | "自选一位见证人" |
| **能力取人** | 调用一个插件能力动态算人（高级用法，通常由开发者配好给你选） | "按值班表取今天的值班经理" |

**解析顺序**：多条规则并集 → 剔除离职者 → 智能去重（§5.1）→ 若最终没人则走空兜底（§5.2）。

### 5.1 智能去重：审批人是发起人自己 / 刚审过怎么办

两个独立开关，每个节点可改（都有推荐默认值）：

**① 候选人就是发起人本人时**（selfPolicy）：

- **自动通过**（默认）：本站直接过，留一条"自动通过"痕迹——自己不用审自己；
- **照常审**：发起人也得亲自点一次同意（如廉政承诺场景）；
- **转直属主管**：换成发起人的直属主管来审。

**② 候选人在上一步刚办理过时**（repeatPolicy）：

- **连续去重**（默认）：这一站的人 ∈ 上一站实际办理人 → 自动通过（连着两级都是同一个经理时，第二级自动过，且会链式传导）；
- **全程去重**：整条流程里已经办理过的人，后面一律自动通过；
- **不去重**：出现几次审几次。

自动通过**不是跳过**：任务照常生成、直接记为已完成，进度图完整可见，痕迹里写明"自动通过"。

### 5.2 空兜底：解析出来一个人都没有怎么办

比如"直属主管"取人，但发起人是老板、没有上级。每个人任务节点必配一个兜底策略（有默认值）：

- **自动通过**：没人审就当过了；
- **转管理员**（默认）：送给实例管理员处理；
- **指定人**：兜底送给指定的某个人；
- **挂起**：流程停住并通知流程管理员来处理。

## 6. 多人审批：四种模式 + 否决策略

一站解析出多个人时，怎么算"这站过了"？在节点的**多人模式**里选：

| 模式 | 规则 | 场景 |
|---|---|---|
| **或签**（默认） | 一张任务多个候选人，**谁抢先认领谁办**，一人同意即过 | 客服池、行政池，"有一个人处理就行" |
| **会签** | 每人一张任务，**全部同意**才过，最后一个同意的推动流转 | 重大事项需全体一致 |
| **依次** | 按顺序一个审完轮下一个 | 逐级把关且有先后 |
| **比例签** | 同意人数达到设定百分比即过（如 60%） | 评审、评议投票 |

**否决策略**（有人点"不同意"时怎么办）：

- **即时生效**（默认）：一人否决立即按节点配置的后果收场（终止/退回），其他人的任务自动撤销。比例签下有个聪明的修正——**数学短路**：只有当反对票多到"同意在数学上不可能达标"时才收场，否则只记录这一票反对，继续等其他人。
- **等全员**：收齐所有人的表态再汇总裁决，中途的否决只记录不发作。适合"想一次拿到所有反对理由"的评审场景。

**认领机制**（或签的配套）：多候选人的任务需要先**认领**再办理——认领即独占，其他人会看到"已被 ×× 认领"；认领后没提交可以释放，让别人接手。打开办理面板会自动尝试认领。

## 7. 时效：超时提醒与超时升级

在人任务节点的"超时"设置里：

- **超时提醒**：任务停留超过 N 小时，给相关人发提醒；可设"之后每 M 小时再提醒一次"。任务办完提醒自动停。
- **超时升级**：提醒之外的强制动作，超过设定时限后自动执行以下之一：
  - **自动通过** / **自动拒绝**（按节点的不同意后果执行）；
  - **自动转交**：转给指定的人；
  - **仅打标**：只在流程痕迹里标记"已超时"，供事后追责，不干预流转。
- **通知目标**：提醒发给谁——当前办理人 / 上一步办理人 / 发起人，可多选。

另有**自动办理**（与超时无关）：节点上配一个条件表达式，任务创建时如果条件已满足就直接自动通过本站。典型："报销金额 < 1000 元，部门经理这一级免审"。

## 8. 连线与路由：流程怎么决定往哪走

### 8.1 条件分支

一个节点拉出多条线时（并行-分除外），构成**排他分支组**：按顺序逐条判断条件，**命中第一条就走它，其余不走**。

- 每条分支线配一个**条件**（如 `请假天数 > 3`），用平台统一的条件编辑器写（引导模式点选拼装，或表达式模式手写）；
- 条件里可以用单据字段、发起人信息、流程变量；
- 可以给分支线配"不成立提示语"，帮办理人理解为什么走了另一条路。

### 8.2 默认线（兜底）

每个分支组**必须**指定一条"默认分支"：所有条件都不成立时走它。这是发布校验强制项——没有兜底的分支组会让流程在"哪条都不满足"时卡死，所以不允许发布。快捷用法：把默认线直接指到"结束"，等价于"其余情况→结束"。

### 8.3 结束的三种方式（都合法）

1. **显式画一个结束节点**，把线连过去；
2. **什么都不画**：没有后续节点的末端，发布时自动补一个隐式结束（画布上不显示，进度图会体现）；末端有个抄送节点也一样成立（抄送发完即结束）；
3. **分支上"其余情况 → 结束"**：默认线直指结束。

**唯一不合法的是"意外的无路可走"**：运行中如果一个节点的条件全不中又没有默认线（老版本数据才可能出现），流程会报错停住而不是静默通过——意外通过比卡住报错危险得多。

### 8.4 回头路？

设计器**不允许画环**（从下游节点连回上游会被校验拦下）。"打回重来"不靠画环实现，而是靠运行时的**退回**操作（§10.3）——这样进度图永远清晰，也不会出现死循环。

## 9. 发布与版本

### 9.1 发布校验：常见报错对照

| 报错 | 意思 | 怎么改 |
|---|---|---|
| 缺少开始节点 / 开始节点只能有一个 | 图上没有 start 或有多个 | 保留恰好一个 |
| 存在不可达节点 | 某节点没有任何入线，从开始走不到 | 连上它或删掉它 |
| 流程图存在回路 | 画了环 | 拆掉回头线，用运行时"退回"替代 |
| 条件分支组必须选择兜底去向 | 分支组没有默认线 | 指定一条默认分支（或"其余情况→结束"） |
| 分支连线缺少条件 | 分支组里有既没条件也非默认的线 | 补条件或设为默认 |
| 一组分支只能有一条默认线 | 默认线配了多条 | 只留一条 |
| 审批节点未配置审批人 / 未配置按钮 | 人任务缺关键配置 | 补齐参与者规则和按钮 |
| "不同意"按钮未配置动作 | reject 按钮没选后果 | 选终止/退回上一步/退回发起人 |
| 比例签阈值无效 | 比例不在 1–100 | 改成合法百分比 |

错误列表可点击定位到具体节点/连线。

### 9.2 版本规则：发布即冻结

- 每次发布生成一个**递增版本号**，发布后的版本**永不可变**；
- **在途实例钉住发起时的版本**：单据是按 2.0 提交的，就按 2.0 的图纸走完，哪怕你后来发了 3.0——改图不影响已经在路上的单子；
- **文案小改即时生效**：节点名称、按钮文字、提示语这类不改变路线的修改，可以直接对已发布版本原地生效，不用发新版本；
- **在途同步（换轨）**：确实需要让在途单据改走新版本时，管理员可对指定实例执行"同步到新版本"——系统会做**锚点校验**（该实例当前停留的节点必须在新版本里还存在且类型没变），校验不过拒绝同步，防止单据换轨后无路可走。支持批量，操作入审计。

## 10. 流程办理：作为办理人/发起人你能做什么

### 10.1 任务中心

左侧菜单的**任务中心**是全局工作台（跨所有应用聚合），四个页签：

- **待办**：等你处理的任务（角标显示数量）；
- **已办**：你处理过的；
- **抄送**：抄送给你的知会件（可批量已阅）；
- **我发起**：你提交的单据都走到哪一步了。

点任一行进入**办理面板**：左侧是只读的单据全文，右侧是流程进度图（走过的、正在办的、被退回的节点一目了然），底部是操作区（按钮 + 意见框 + 暂存）。

### 10.2 基本操作

| 操作 | 说明 |
|---|---|
| **认领 / 释放** | 多候选任务先认领再办（打开面板自动尝试认领）；没提交前可释放给别人 |
| **同意 / 完成** | 表态放行，流程向下走；意见按节点配置可选/必填 |
| **不同意** | 按节点配置的后果执行：整单终止 / 退回上一步 / 退回发起人 |
| **暂存** | 意见写了一半先存着，任务保持在你名下，下次打开接着写 |
| **批量审批** | 待办里勾选多条一键同意；要求必填意见/必填补填的任务会被自动排除并提示原因 |

### 10.3 退回与重新提交

- **退回上一步**：单据回到上一站，由**原办理人**重办（新生成一条任务，历史痕迹不动）；
- **退回发起人**：单据回到发起人手里修改重提；
- **退回指定节点**：沿"已经走过的路径"退到指定一站（不能退到没走过的节点）；
- **清空重填**：节点可以配置"退回时哪些字段要清空重填"，被退回的人打开会看到提示；
- **重提方式**：发起人修改后重新提交，默认**直达**当初退回的那一站（中间已经审过的不再重复审）；节点也可配置为**重走全程**。

退回受并行守卫约束：并行支线内不能跨支线退（§4.6）。

### 10.4 发起人的特权

- **撤回**：单据提交后、**下游还没有任何人实际处理**（没人办结、没人认领）时，发起人可以撤回——单据回到未进流程状态，改完可重新提交。一旦有人动过就不能撤了（从严设计，避免"审到一半单子消失"）。
- **催办**：手动提醒当前办理人，带防骚扰节流（同一任务短时间内不能反复催）。

### 10.5 更多协作能力

以下能力引擎已经完整支持（部分管理界面在后续版本逐步开放入口）：

| 能力 | 说明 |
|---|---|
| **转办** | 把自己的任务换成另一个人来办，痕迹记"由 B 代 A 办理" |
| **委托代理** | 规则而非单次动作：设置"某时间段内我的审批全部（或某条流程的）自动交给某人"——休假必备。命中委托时任务直接生成给受托人，痕迹记录代理关系 |
| **加签** | 动态调整会签群：**前加签**（先让加签人审，审完回到我）、**后加签**（我审完先给加签人，再继续流转）、**并加签**（拉人和我一起会签）、**减签**（移除还没办理的会签人） |
| **管理员干预** | 流程管理员可强制终止实例、强制转交任务（痕迹记录管理员身份），处理挂起/异常的单据 |

### 10.6 痕迹与追溯

流程的每一步——发起、认领、同意、拒绝、退回、转办、催办、自动通过——都会追加一条**不可篡改的流水**（只能追加，不能改不能删），带操作人、时间、意见。进度图 + 流水时间线 = 完整审计链。

## 11. 常见问题（FAQ）

**Q：提交后发现单据填错了怎么办？**
A：下游还没人处理 → 直接**撤回**改完重提。已有人处理 → 联系当前/后续审批人点**不同意-退回发起人**，或等流程走完拒绝后再改。

**Q：为什么我的表单上没有"提交"按钮？**
A：该模板没有绑定任何启用的流程。绑定流程后提交入口才会出现；未绑定流程的模板走普通"保存"。

**Q：审批中的单据，办理人能改内容吗？**
A：默认整单只读。只有当前节点的**字段权限**里明确列为"可编辑"的字段，当前办理人才能改；其他人、其他字段一律只读。

**Q：同一个人在两级审批里都出现，要审两次吗？**
A：默认不用——"连续去重"会让第二级自动通过并留痕。想让他审两次，把节点的去重策略改为"不去重"。

**Q：审批人离职了，单子会卡死吗？**
A：不会走丢：取人时离职者自动剔除；如果剔完没人了，走节点的**空兜底**（默认转管理员）。已经生成的存量任务可由管理员强制转交。

**Q：改了流程图，之前提交的单子会乱吗？**
A：不会。在途单据钉住提交时的版本走完。确需换轨用管理员的"同步到新版本"，有锚点校验兜底。

**Q：怎么删掉/停用一条流程？**
A：解除/停用模板绑定即可让新提交不再走它；在途实例不受影响继续走完。

**Q：审批通过的单据还能删除吗？**
A：审批中（processing）的单据禁止删除；已通过/已拒绝的单据按模板的常规权限处理。
