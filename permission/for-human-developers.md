# 权限框架（代号 卡侬/Canon）业务原理与插件接入指南 —— 面向开发者

> 读者：要开发 Kimpo 插件、或要把自己的业务对象纳入平台权限管控的**人类工程师**。
> 读完你会明白：卡侬(Canon)是什么、部门/用户/角色/应用角色四类主体怎么协作、一条授权是怎么算出"你能不能做"的，以及你的插件**为什么几乎什么都不用做**就已经被正确管控。
> 配套：给大模型/AI 编程助手看的结构化版本在同目录 `for-llm-agents.md`（把它喂给你的 AI 助手，它就懂规矩）。

## 1. 卡侬(Canon)是什么：一扇门、一张名单、一个算式

Kimpo 的权限框架（代号 卡侬/Canon）回答一个问题：**"这个用户，对这个业务对象，能做什么？"**

它由三样东西组成：

- **一张名单（授权表）**：管理员配置的授权项，每条 = "把某些权限，授予某个主体，作用在某个客体上"。存在应用库里，随应用备份/分发。
- **一个算式（合并裁决）**：请求到来时，把用户展开成他的全部身份（本人 + 所属部门 + 绑定的角色），凡命中名单的授权项全部取出，**并集合并、就宽不就窄**，得到"有效权限"。
- **一扇门（执法层）**：宿主在数据入口处（列表查询、保存、删除、导入导出、填报会话建立）统一放行或拦截。**门在宿主，不在插件**。

三条底层原则，记住它们就理解了一切行为：

1. **白名单默认**：没有任何授权项命中 ⇒ 一律不可见/不可操作（特权身份除外）。新模板发布后普通用户看不到，是特性不是 bug——先配权。
2. **并集取最宽**：多条授权同时命中时，操作权限取"或"，数据范围取最宽档。**没有"拒绝"记录**——收权 = 删授权或缩载荷，不存在优先级迷宫。
3. **后端是唯一权威**：前端隐藏菜单、置灰按钮只是体验优化；就算前端被绕过，宿主的门照样拦。

**特权绕过**：三种身份不查名单直接全权——超级管理员（super_admin）、应用设计者（app_designer）、该应用的应用管理员（app_admin）。所以管理员和设计者永远不会被自己配的权限锁在门外。

## 2. 四类主体：部门、用户、两层角色的关系

一条授权可以授给四类主体（grantee），它们只是"圈人的四种方式"，在裁决里完全平等：

| 主体类型 | 是什么 | 圈到谁 |
|---|---|---|
| **用户（user）** | 具体一个人 | 就这个人 |
| **部门（org_unit）** | 组织树上的一个节点 | 该部门全部成员（主部门+副部门都算）；开了 `include_children` 则整棵子树的成员都算 |
| **实例角色（role）** | **平台级**角色树上的角色（如"财务""区域经理"），一次定义、所有应用可用 | 绑定了该角色的用户 |
| **应用角色（app_role）** | **应用自己**的角色树（如本应用的"审批人""填报员"），随应用走 | 在本应用被分配了该角色的用户 |

**两层角色为什么并存**：实例角色描述"这个人在公司里是什么"（跨应用复用）；应用角色描述"这个人在这个应用里干什么"（应用自治，应用管理员就能建，不用求平台超管）。**关键设计**：应用角色的定义存在应用库（随应用分发），但"谁拥有该角色"的分配关系存在实例库（不随应用包外泄）——所以把应用分发给另一家实例后，权限设置全在，只需给新实例的用户分配角色，权限立即生效。

**一个用户的身份集合**（裁决第一步）：

```
identities(用户) = { 本人 }
                ∪ { 他绑定的全部实例角色 }
                ∪ { 他在本应用绑定的全部应用角色 }
                ∪ { 他所属的全部部门 }
                ∪ { 这些部门的祖先部门中，开了 include_children 的授权所指的那些 }
```

任何一条授权的主体命中这个集合，就参与合并。部门进人、角色绑人，**权限自动生效**，不需要重新配权。

## 3. 三层客体与权限载荷

权限作用的对象（客体）分三层，逐层收敛——上层不通过，下层不再评估：

| 层 | 客体 | 载荷 | 语义 |
|---|---|---|---|
| 1 | **应用（app）** | 允许访问（1 个开关） | 能不能看到/进入这个应用 |
| 2 | **菜单（menu）** | 允许查看（1 个开关） | 应用导航树上这个节点显不显示（menu 客体 = 应用树节点，object_id 用 node_key） |
| 3 | **模板（template）** | 操作矩阵 + 数据范围 | 核心层，见下 |

**模板层载荷分两段**：

- **操作矩阵**（7 个布尔开关）：`create` 新建单据、`view` 查看历史、`modify` 修改历史、`delete` 删除、`import` 导入、`export` 导出、`print` 打印。
- **数据范围**（行级，挂在 view/modify/delete 三个操作上各一份档位）：

| 档位 | 能看/改/删哪些行 |
|---|---|
| `all` | 全部数据 |
| `dept_tree` | 归属部门在我的部门子树内的行 |
| `dept` | 归属部门 = 我的部门的行 |
| `self` | 归属人 = 我的行 |

**行的"归属"是什么**：每条主表记录带两个可变归属列——`owner_id`（归属人，初始=创建人，交接/转移时可改）和 `owner_org_id`（归属部门，初始=创建人当时的主部门，快照语义：人调岗了历史单据留在原部门）。行级裁决按 owner 列，**不按创建人**——所以单据交接给别人后，接手人按"本人"档能看到、原创建人看不到，这是设计意图。明细/扩展表的行跟随其主记录的可见性，不单独授权。

**注意两个"部门"不要混淆**：授权主体里的"部门 + include_children"决定**谁拥有权限**；数据范围里的 `dept/dept_tree` 决定**能看哪些行**。前者是圈人，后者是圈数据。

## 4. 授权模式：一条授权长什么样

```
授权项 = 主体(grantee_type + grantee_id [+ include_children])
       × 客体(object_type + object_id)
       × 载荷(操作矩阵 + 三个数据范围档位)
```

配置入口（管理员视角，你开发时用来造测试数据）：

- **应用设置 › 权限管理**：左侧客体树（应用/菜单/模板）任选一个客体，右侧看它的全部授权项、添加/编辑/删除授权；
- **模板版本管理页 › 权限设置卡片**：直接管理单个模板的授权；
- 添加授权弹窗三页签：部门树多选（每项可勾 include_children）/ 角色（全局角色+本应用角色两分组）/ 用户搜索多选——一次可授多个主体同一载荷。

**合并示例**：张三属"销售一部"（部门授权：view=✓, view_scope=dept），又绑了实例角色"审计"（角色授权：view=✓ view_scope=all, export=✓）。合并结果：view=✓（范围取宽 → all）、export=✓。后来把"审计"角色从张三身上解绑——他自动回落到只看本部门，无需改任何授权配置。

## 5. 你的插件需要做什么？——几乎什么都不用

这是卡侬对插件开发者最重要的承诺：**插件与权限零耦合**。你不查授权表、不判断用户身份、不实现任何权限逻辑。原因是门都设在宿主的数据面上：

| 你的插件行为 | 宿主替你做的事 |
|---|---|
| 动作被触发（如取数、写值） | 触发它的用户已经过了操作闸门——你的动作能被调起，本身就说明有权 |
| 经 `host.Mirror()` 读会话数据 | 会话建立时已按 FormMode 预检过（新增需 create 权、打开历史需 view 权），Mirror 里就是这个用户可见宇宙内的数据 |
| 经 `host.Query()` 反向查询 | 宿主编译执行时套用查询者的权限（模板可用性校验） |
| 经 `host.Record()` 写记录 | 记录级闸门（modify/delete + 行级范围）在宿主写入口校验，越权记录对你表现为"不存在" |
| 写数据带 `write_grant` | 那是动作级写授权（哪个动作能写哪张表），与用户级权限双层叠加，你只管原样透传 |

**你唯一可能想主动感知权限的场景**：插件的**前端 Surface** 想按当前用户权限定制界面（比如隐藏一个"删除"按钮）。Surface 运行在宿主前端上下文里，可直接同源调用：

```
GET /api/v1/apps/{appId}/effective-permission
→ { "is_privileged": bool,            // true=特权全权，直接短路，别逐项比对
    "app_access": bool,
    "visible_menu_keys": ["node_key", …],
    "templates": { "<templateId>": {
        "create": bool, "view": bool, "modify": bool, "delete": bool,
        "import": bool, "export": bool, "print": bool,
        "view_scope": "all|dept_tree|dept|self|", … } } }
```

记住：这只是体验优化。**别**基于它做安全判断——后端的门才是权威。

**红线**（违反即打回）：
- 不要自建权限表/权限判断逻辑——平台权限只有卡侬一套；
- 不要试图绕过行过滤（比如自拼 SQL 查物理表——这同时违反数据通道红线 R1）；
- 不要缓存 effective-permission 超过一个页面会话——授权变更后宿主侧 60 秒内生效，你的长缓存会造成界面与实际权限漂移。

## 6. 想把你的业务对象纳入卡侬管控？——扩展侧指南

如果你在开发的东西有自己的"业务对象"想被统一赋权（比如一种新的可授权资源），按优先级考虑三条路：

1. **首选：挂到现有客体上**。如果你的对象呈现在应用导航树上（page/history 类插件的页面节点），它天然是 menu 客体，管理员在权限管理里就能对它授"允许查看"——你什么都不用做，宿主的树过滤已覆盖它。
2. **对象与模板绑定**：如果你的功能围绕某个模板展开（取数、审批、视图），复用该模板的操作矩阵语义——用户对模板有什么权，对你的功能就有什么权。
3. **新客体类型（需要宿主侧开发，与平台团队协作）**：卡侬的客体是开放枚举（object_type 现为 app/menu/template），载荷操作位可扩展（加一个布尔位不动模型）。宿主侧的接入模式是"小接口 + 注入"：你的模块定义自己的最小权限接口（如 `CanAccessRecord(ctx, user, object, op) bool`），由宿主装配层注入卡侬的适配器——模块不直接依赖卡侬包，保持低耦合。裁决入口是 `ComputeForUser(实例, 应用, 用户)`（内部自带特权短路与 60s 缓存）和 `ResolveRowPredicate(...)`（把数据范围档位翻译成行过滤谓词）。

## 7. FAQ

**Q：管理员没给任何人配权，谁能用模板？**
A：只有 super_admin / app_designer / 该应用 app_admin。白名单默认——发布后配权是必要动作。

**Q：用户对模板有数据权限，但导航里看不到它？**
A：模板授权 ≠ 菜单授权。要它在导航可达，还需对其所在菜单节点授"允许查看"（或直接对模板客体授权后走直链）。这是有意分层：数据权和导航可见分开控制。

**Q：越权访问一条记录，会得到 403 还是 404？**
A：404。存在但无权的记录与不存在的记录表现一致——不泄露存在性。你的插件对"记录不存在"的处理天然覆盖了越权场景。

**Q：权限变更多久生效？**
A：授权项增删改：立即（宿主主动失效缓存）。角色绑定/部门成员变更：最迟 60 秒（缓存 TTL 到期）。

**Q：能配"拒绝某人"吗？**
A：不能，卡侬无 deny 语义。收权 = 删掉/缩小他命中的授权。真要排除个别人，把授权主体从大部门换成更精确的子部门/角色/用户列举。
