# 艾莉亚(Aria)业务原理与 SDK 对接指南 —— 面向插件开发者

> 读者：要开发 Kimpo 插件、需要读写填报表单业务数据的**人类工程师**。
> 读完你会明白：艾莉亚(Aria)是什么、为什么必须经过它、以及怎么用 SDK 十分钟接上。
> 配套：给大模型/AI 编程助手看的结构化版本在同目录 `for-llm-agents.md`（把它喂给你的 AI 助手，它就懂规矩）。

## 1. 艾莉亚(Aria)是什么：一个账本的故事

想象填报一张表单时，系统里有三个角色：

- **账本（艾莉亚(Aria) Mirror）**：宿主进程里的一份内存权威数据。表单上每个业务字段"此刻到底是什么值"，**以它为准**。保存落库时读的是它，动作取值时读的是它。
- **前台（读写统一入口）**：账本从不让任何人直接翻页。所有读写都经过前台——它给**每个填报会话排一条队**，写有校验、读有屏障，保证任何人拿到的都是"当时那一刻的真实数据"。
- **显示屏（editor，表格编辑器）**：负责把账本画给用户看、承接用户敲键盘，并且是**公式计算内核**（因为公式会用到"非业务字段"的辅助单元格，那些格子只有它有）。但它**不是权威**——它算出来的结果也要交回前台记账。

你的插件（取数、写值、任何业务动作）是账本的**对等读写方**：跟编辑器平起平坐，都排前台的队，谁也没有特权。

**一句话记住**：*业务数据只有一个真相（账本），进出只有一扇门（前台/SDK），显示屏只管画和算。*

## 2. 为什么必须这样？——没有艾莉亚(Aria)的世界

这套架构不是洁癖，是从真实事故里长出来的。曾经（旧模型 B-1）插件写数据要"拜托编辑器代写"，编辑器再异步把整屏内容刷回后台，结果：

1. **回声覆盖**：插件刚写入 A，编辑器一次定时的整屏回刷携带旧值把 A 冲掉——用户看着数据"自己变回去了"。
2. **脏读半算值**：公式还没算完，另一个动作已经把中间值读走并写进了别的字段。
3. **乱序**：多个写方（用户敲键盘、取数动作、控件填充）各写各的，落库结果取决于谁的网络包先到。

艾莉亚(Aria)把这三类问题**在架构上**消灭了：权威只有一份、写入有序、公式没算完就挡住依赖它的读写、编辑器的回刷带"来源过滤"不再回声。**而你的插件什么都不用做**——这些保障内建在你调用的每一个 SDK 方法里。

## 3. 一次写入的生命周期（你调用 WriteField 之后发生了什么）

```
你的插件                     宿主（前台+账本）                        编辑器/浏览器
   │  WriteField(…grant)        │                                      │
   ├───── SDK gRPC ────────────▶│ ① 验 grant（这个动作允许写这张表吗）    │
   │                            │ ② 写前校验：公式字段？只读会话？——拒    │
   │                            │ ③ 排队+等结算（前面有公式在算就等）      │
   │                            │ ④ 记账本，writeSeq++                  │
   │                            │ ⑤ 通知编辑器 ─────────────────────────▶│ 更新单元格
   │                            │ ⑥ 若触发公式：编辑器算 → 结果回前台记账  │ 重算公式
   │◀───── 返回 ────────────────│ ⑦ 屏障放行后续操作                     │ 推送浏览器渲染
```

注意第 ⑤⑥ 步：**渲染和公式是宿主替你协调的**。你不需要（也不允许）自己去碰编辑器。

## 4. 你自动获得的四项保障

| 保障 | 内容 | 对你意味着 |
|---|---|---|
| **写前校验** | 公式派生字段拒写、查看态会话拒写、失效会话拒写，全在宿主门口 fail-closed | 非法写不会污染数据，错误当场返回，账本干净 |
| **结算屏障** | 你的写触发了公式，宿主保证"公式算完才放行后续业务操作" | 多步动作（取数A→用A的结果取数B）天然有序，不会读到半算值 |
| **渲染自动同步** | 宿主自动通知编辑器更新单元格并推送浏览器 | 写完即所见，你零渲染代码 |
| **防回声** | 你写入的值不会被编辑器旧值回刷覆盖；用户后续手改又能正常覆盖你 | 值的"最终归属"永远符合直觉 |

## 5. 动手：SDK 对接

依赖：`github.com/kimposoft/kimpo-plugin-sdk`（下称 SDK）。完整插件骨架请用脚手架 `new-action-plugin` 生成，这里只讲数据交互。

### 5.1 拿句柄与上下文

动作被触发时，SDK 把宿主句柄放在 ctx 里；会话与写授权在触发上下文（`query_context`）里：

```go
host := plugin.HostFrom(ctx)          // 宿主句柄（nil 判空防御）
mirror := host.Mirror()               // 艾莉亚(Aria)客户端

// 触发上下文（宿主随动作分发注入，原样取用）：
sessionID, _ := qc["sheet_session_id"].(string) // 目标填报会话
writeGrant, _ := qc["write_grant"].(string)     // 本次调用的写授权令牌
```

`writeGrant` 是宿主为**这一次动作调用**签发的（call-scoped）：只覆盖动作配置里声明的目标表/字段。**原样透传即可，不要存储、不要跨动作复用**——过期即废，伪造无效。

### 5.2 读

```go
// 单字段。waitSettled=true：等公式全部结算再返回（要权威值就传 true）
found, tv, err := mirror.ReadField(ctx, sessionID, tableID, "main:0", fieldID, true)

// 整表快照（做内存 join/排序用）；行序按显示序
rows, err := mirror.ReadTable(ctx, sessionID, tableID)

// 枚举明细行 rowKey（"row:0","row:1",…）
rowKeys, err := mirror.ListDetailRows(ctx, sessionID, tableID)
```

**rowKey 约定**：主表恒为 `main:0`；明细/扩展表为 `row:N`（稳定身份键，删行重排不漂移）。

### 5.3 写

值的线格式恒为 JSON `{"raw": 业务值, "text": "显示文本"}`：

```go
tv, _ := json.Marshal(map[string]any{"raw": 42, "text": "42"})

// 单字段
_, err := mirror.WriteField(ctx, sessionID, tableID, "main:0", fieldID, tv, writeGrant)

// 批量（推荐：一批 = 一个屏障窗口 = 一帧渲染，比逐字段快且原子感更好）
results, err := mirror.WriteFields(ctx, sessionID, fields, writeGrant)
```

**批量写必须逐条检查 `results[i].Err`**：批里某个字段单独写失败（如混入公式字段被宿主拒写）不会让整批报错，而是透出在对应结果项上。两类语义按 `AppliedRevision` 区分：

- `Err != ""` 且 `AppliedRevision == 0`：**该字段权威未写**（如 computed 拒写）——值没进账本，按 5.5 错误表处置；
- `Err != ""` 且 `AppliedRevision > 0`：权威已写、仅渲染通知失败——按"写成功"继续（见 5.5 最后一条）。

```go
for _, r := range results {
    if r.Err != "" && r.AppliedRevision == 0 {
        // 该字段没写进去（比如它是公式字段）——处理或上报
    }
}
```

### 5.4 明细行生命周期

```go
rowKey, err := mirror.AppendDetailRow(ctx, sessionID, tableID, writeGrant)      // 追加一行 → 拿 rowKey 再写字段
err = mirror.RemoveDetailRow(ctx, sessionID, tableID, rowKey, writeGrant)       // 删一行
err = mirror.ReorderDetailRows(ctx, sessionID, tableID, orderedKeys, writeGrant) // 整表重排
```

追加前先看看有没有可复用的空行（`ReadTable` 后挑全空行），避免凭空增行。

### 5.5 错误处理（四条语义背熟）

| 你收到 | 含义 | 正确反应 |
|---|---|---|
| `computed_readonly`（批量时在 `results[i].Err` 且 `AppliedRevision==0`） | 目标是公式派生字段 | 别写它——它的值由公式算，你写输入字段就好 |
| `session not writable` | 查看态（view）会话 | 只读场景，放弃写 |
| `mirror session not found` | 会话未开/已回收 | 会话失效，终止本次动作，勿重试 |
| **渲染通知失败**（单字段 = `*plugin.RenderNotifyError`，用 `errors.As` 判定；批量 = `results[i].Err` 非空且 `AppliedRevision>0`） | **值已进账本、会随保存落库**，只是界面暂时没刷 | 按"写成功"继续业务流程，最多记条 warn |

最后一条最容易犯错：**不要**因为渲染通知失败就回滚你的业务流程——权威已经写入了。

### 5.6 查询（读别的模板/数据表）

跨模板取数不要自己碰数据库。用 `host.Query()` 提交表达式模型（或 KimpoSQL 文本），宿主负责编译成 SQL、鉴权、执行，返回行集给你在内存里加工。详见取数插件参考实现与《查询执行概念与业务逻辑》。

## 6. 红线（PR 审查会卡的五条）

1. ❌ **禁止直连编辑器**：不 import editorpb、不调编辑器 RPC、不经 WS 命令写业务值。
2. ❌ **禁止裸 SQL** 读写 `app_table_*` 业务表——查询一律走 `host.Query()`。
3. ❌ **禁止囤积/复用 writeGrant**（它是一次性调用凭证）。
4. ❌ **禁止把编辑器/浏览器当数据源**——要权威值就 `ReadField(waitSettled=true)`。
5. ❌ （编辑器类插件）禁止把宿主写入的值原样回投——必须实现来源过滤。

## 7. FAQ

**Q：我写公式字段为什么被拒？**
它是"算出来的"字段（computed），账本对它只收编辑器的计算结果。你应该写它的**输入**字段，公式会自动重算。

**Q：写成功了但界面没变？**
分两种：返回了"渲染通知失败"→ 值已在账本，刷新/重开表单即见；什么错都没有但看不见 → 检查你写的 tableID/rowKey/fieldID 是否真的绑定在当前表单上。

**Q：我想做一个新的编辑器类插件（自己渲染表单）？**
除本文能力外你要多承担三个契约：开会话时声明公式字段清单、把用户编辑经 `SyncFields`/`ReportCalcState` 有序上行、接收宿主渲染通知且**无否决权**。详见宿主仓 `docs/dev-guide/plugin-business-data-rw-spec.md` §2.2。

**Q：性能——我要写几百个格子？**
用 `WriteFields` 分批（每批同表），一批一个屏障窗口；不要几百次 `WriteField` 单调。
